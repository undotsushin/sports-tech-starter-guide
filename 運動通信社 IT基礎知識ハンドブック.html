1. Webサイト構造とパーツ集

🎯 なぜ必要か

クライアントやエンジニアとの会話で「ナビを変更して」「フッターに追加して」といった指示を正確に伝えるため。共通言語として必須の知識です。

1.1 基本構造パーツ

ヘッダー（Header）

  • 役割: サイトの最上部にある共通エリア
  • 含まれるもの: ロゴ、メインナビゲーション、検索窓、ログインボタン
  • 実務での重要性: ブランディングとユーザー導線の起点

フッター（Footer）

  • 役割: サイトの最下部にある共通エリア
  • 含まれるもの: 著作権表記、プライバシーポリシー、お問い合わせリンク
  • 実務での重要性: 法的情報とサブ導線の配置場所

ナビゲーション（Navigation/Nav）

  • 役割: サイト内の主要ページへのリンク集合
  • 種類: グローバルナビ、ローカルナビ、パンくずナビ
  • 実務での重要性: ユーザーの迷子を防ぐ重要な導線設計

サイドバー（Sidebar）

  • 役割: メインコンテンツの横に配置される補助エリア
  • 含まれるもの: 関連記事、広告枠、SNSボタン
  • 実務での重要性: 収益化とエンゲージメント向上の場所

1.2 コンテンツパーツ

CTAボタン（Call To Action）

  • 役割: ユーザーに特定のアクションを促すボタン
  • 例: 「今すぐ申し込む」「資料ダウンロード」「お問い合わせ」
  • 実務での重要性: コンバージョン率に直結する重要要素

フォーム（Form）

  • 役割: ユーザーからの情報入力を受け取る仕組み
  • 種類: お問い合わせフォーム、会員登録フォーム、アンケートフォーム
  • 実務での重要性: リード獲得の最重要ポイント

パンくずリスト（Breadcrumb）

  • 役割: 現在位置をユーザーに示すナビゲーション
  • 例: トップ > カテゴリ > 記事タイトル
  • 実務での重要性: SEOとユーザビリティの向上

理解度チェック問題 1-1

  1. ユーザーが迷子にならないようにするために最も重要なパーツは？
  2. コンバージョン率に直接影響するパーツは？
  3. 法的情報を配置する場所として適切なのは？
  4. SEO効果も期待できるナビゲーションパーツは？

解答: 1. ナビゲーション 2. CTAボタン 3. フッター 4. パンくずリスト

2. Web技術基礎

🎯 なぜ必要か

システム構築の予算・期間を適切に見積もり、技術的制約を理解してプロジェクト管理を適切に行うため。

2.1 ブラウザの基礎知識

ブラウザとは

  • 定義: Webページを表示するためのアプリケーション
  • 主要ブラウザ: Chrome、Safari、Firefox、Edge
  • 実務での重要性: ブラウザによって表示が異なる可能性があるため、対応範囲を決める必要がある

ブラウザ対応の考え方

  • モダンブラウザ: 最新の技術に対応
  • レガシーブラウザ: 古い技術のみ対応（IE11など）
  • 実務での判断: ターゲットユーザーの利用ブラウザに合わせて対応範囲を決定

2.2 ドメインとURL

ドメイン（Domain）

  • 定義: インターネット上の住所のようなもの
  • 例: sportsbull.jp、google.com
  • 種類:
    • トップレベルドメイン（.jp、.com）
    • サブドメイン（blog.sportsbull.jp）

URL（Uniform Resource Locator）

  • 構造: https://www.sportsbull.jp/news/article/123
  • 要素:
    • プロトコル: https://
    • ドメイン: www.sportsbull.jp
    • パス: /news/article/123

2.3 サーバーとホスティング

サーバー（Server）

  • 役割: Webサイトのデータを保存・配信するコンピュータ
  • 種類:
    • 共有サーバー: 複数のサイトで共用
    • 専用サーバー: 1つのサイト専用
    • クラウドサーバー: AWS、Google Cloud等

SSL証明書

  • 役割: データの暗号化とサイトの信頼性証明
  • 見分け方: URLが https://で始まる
  • 実務での重要性: SEOとユーザーの安心感に影響

2.4 レスポンシブデザイン

定義

  • PC、スマホ、タブレットなど異なる画面サイズに対応したデザイン
  • 必要性: 現在はスマホユーザーが過半数
  • 実装: CSSのメディアクエリを使用

理解度チェック問題 2-1

  1. URLの https://部分を何と呼ぶ？
  2. サイトが暗号化されているかどうかはどこで判断できる？
  3. スマホとPCで同じサイトが最適表示される技術は？
  4. インターネット上の住所に例えられるものは？

解答: 1. プロトコル 2. URLがhttps://で始まる 3. レスポンシブデザイン 4. ドメイン

3. アプリ基礎知識

🎯 なぜ必要か

アプリ企画時の開発コスト・期間・制約を正しく判断し、最適な開発方針を決定するため。

3.1 アプリの種類と特徴

ネイティブアプリ

  • 定義: iOS/Android専用に開発されたアプリ
  • 特徴:
    • アプリストアからダウンロード
    • 端末の機能をフル活用可能
    • オフラインでも動作
    • 開発コストが高い

Webアプリ

  • 定義: ブラウザ上で動作するアプリケーション
  • 特徴:
    • インストール不要
    • URL でアクセス
    • 開発コストが低い
    • 端末機能の制限あり

ハイブリッドアプリ

  • 定義: Webとネイティブのいいとこどりをしたアプリ
  • 特徴:
    • 一度の開発で複数プラットフォーム対応
    • ストアからダウンロード可能
    • 性能はネイティブより劣る場合が多い

PWA（Progressive Web App）

  • 定義: Webアプリをアプリのように使える技術
  • 特徴:
    • ホーム画面に追加可能
    • オフライン動作
    • プッシュ通知対応

3.2 アプリストアの仕組み

App Store（iOS）

  • 審査期間: 通常1-7日
  • 手数料: 売上の30%（小規模事業者は15%）
  • 特徴: 審査が厳格

Google Play Store（Android）

  • 審査期間: 通常数時間〜1日
  • 手数料: 売上の30%（小規模事業者は15%）
  • 特徴: 審査が比較的緩い

3.3 開発コストの目安

簡単なアプリ（情報表示系）

  • 期間: 2-3ヶ月
  • コスト: 300-500万円

中規模アプリ（SNS、EC系）

  • 期間: 6-12ヶ月
  • コスト: 1000-3000万円

大規模アプリ（ゲーム、複雑なシステム）

  • 期間: 1年以上
  • コスト: 5000万円以上

理解度チェック問題 3-1

  1. オフラインでも動作するアプリの種類は？
  2. 開発コストが最も安いアプリの種類は？
  3. App Storeの手数料は通常何%？
  4. ホーム画面に追加できるWebアプリの技術は？

解答: 1. ネイティブアプリ（PWAも正解） 2. Webアプリ 3. 30% 4. PWA

4. 広告・マーケティング技術

🎯 なぜ必要か

広告予算の適切な配分と効果最大化を図り、クライアントに具体的な提案ができるため。

4.1 動画広告の種類

プリロール（Pre-roll）

  • 定義: 動画コンテンツの再生前に流れる広告
  • 長さ: 通常15-30秒
  • 特徴: 高い視聴率、スキップ不可が多い
  • 実務での活用: 認知度向上に最適

ミッドロール（Mid-roll）

  • 定義: 動画コンテンツの途中に挿入される広告
  • 特徴: 視聴継続率が高い、ユーザーは嫌がる傾向
  • 実務での活用: 長尺コンテンツでのマネタイズ

ポストロール（Post-roll）

  • 定義: 動画コンテンツの終了後に流れる広告
  • 特徴: 視聴率は低いが、興味のあるユーザーが残る
  • 実務での活用: 詳細な商品説明に適している

4.2 広告配信システム

アドネットワーク（Ad Network）

  • 定義: 複数の媒体をまとめて広告配信するサービス
  • 例: Google AdSense、Yahoo!広告
  • メリット: 一括管理、スケールメリット

DSP（Demand Side Platform）

  • 定義: 広告主側の広告配信プラットフォーム
  • 機能: ターゲティング、入札自動化
  • 実務での重要性: 効率的な広告運用に必須

SSP（Supply Side Platform）

  • 定義: 媒体側の広告枠販売プラットフォーム
  • 機能: 広告枠の価格最適化
  • 実務での重要性: 収益最大化に重要

4.3 効果測定指標

インプレッション（Impression）

  • 定義: 広告が表示された回数
  • 用途: リーチの測定

クリック率（CTR: Click Through Rate）

  • 計算式: クリック数 ÷ インプレッション数 × 100
  • 目安: ディスプレイ広告で0.1-0.3%

コンバージョン率（CVR: Conversion Rate）

  • 計算式: コンバージョン数 ÷ クリック数 × 100
  • 重要性: 最終的な成果指標

CPA（Cost Per Acquisition）

  • 計算式: 広告費 ÷ コンバージョン数
  • 用途: 獲得効率の評価

理解度チェック問題 4-1

  1. 動画開始前に流れる広告の種類は？
  2. 広告主側の配信プラットフォームを何と呼ぶ？
  3. CTRの一般的な目安は（ディスプレイ広告）？
  4. 獲得効率を測る指標は？

解答: 1. プリロール 2. DSP 3. 0.1-0.3% 4. CPA

5. データ・システム基礎

🎯 なぜ必要か

法的リスク回避とシステム連携の可能性を判断し、安全で効率的なシステム設計を行うため。

5.1 データベース基礎

データベースとは

  • 定義: データを効率的に保存・検索するシステム
  • 種類:
    • リレーショナルデータベース（MySQL、PostgreSQL）
    • NoSQLデータベース（MongoDB、DynamoDB）

API（Application Programming Interface）

  • 定義: システム間でデータをやり取りする仕組み
  • 例: Twitter API、Google Maps API
  • 実務での重要性: 外部サービスとの連携に必須

5.2 クラウドサービス

主要クラウドサービス

  • AWS（Amazon Web Services）: 最大手、豊富なサービス
  • Google Cloud Platform: AI・機械学習が強い
  • Microsoft Azure: Microsoft製品との親和性が高い

クラウドのメリット

  • 初期投資不要
  • スケーラビリティ
  • 高い可用性
  • 自動バックアップ

5.3 セキュリティ基礎

個人情報保護

  • 個人情報保護法: 日本の法律
  • GDPR: EU の規則、日本企業も対象になる場合あり
  • 実務での重要性: 法的リスクとブランドリスクの回避

Cookie規制

  • 定義: Webサイトがユーザー情報を保存する仕組み
  • 規制動向:
    • サードパーティーCookieの廃止予定
    • ユーザー同意の必要性増加
  • 実務での影響: 広告効果測定の精度低下

理解度チェック問題 5-1

  1. システム間のデータ連携に使われる仕組みは？
  2. 最大手のクラウドサービスは？
  3. EUのデータ保護規則を何と呼ぶ？
  4. Webサイトがユーザー情報を保存する仕組みは？

解答: 1. API 2. AWS 3. GDPR 4. Cookie

6. 実践シチュエーション集

🎯 なぜ必要か

実際の業務で即座に適切な判断・行動ができ、クライアントやチームから信頼される対応をするため。

6.1 クライアント対応シーン

シーン1: 「サイトが重い」と言われた時

よくある原因:

  • 画像サイズが大きすぎる
  • サーバーのスペック不足
  • プラグインが多すぎる

対応方法:

  1. 具体的な症状をヒアリング（どのページが、いつ、どのくらい遅いか）
  2. PageSpeed Insights等のツールで測定
  3. 原因に応じた改善提案（画像圧縮、サーバー強化、不要プラグイン削除）

シーン2: 「CVRが悪い」と相談された時

確認ポイント:

  • 現在のCVR数値
  • 業界平均との比較
  • ユーザーの離脱ポイント

改善提案:

  • CTAボタンの色・位置・文言変更
  • フォームの項目数削減
  • ページ表示速度の改善
  • モバイル対応の見直し

シーン3: 「予算を抑えたい」と言われた時

提案できる選択肢:

  • 機能の優先順位付け（MVP開発）
  • 既存テンプレートの活用
  • 段階的リリース
  • オープンソースの活用

6.2 社内エンジニア連携シーン

シーン1: 仕様変更を依頼する時

やってはいけないこと:

  • 「簡単だと思うので」という前置き
  • 期限直前での依頼
  • 曖昧な指示

正しいアプローチ:

  1. 変更理由を明確に説明
  2. 具体的な要件を文書で共有
  3. 工数見積もりを依頼
  4. 優先度を相談

シーン2: バグを発見した時

レポートに含めるべき情報:

  • 発生環境（OS、ブラウザ、デバイス）
  • 再現手順
  • 期待する動作
  • 実際の動作
  • スクリーンショットまたは動画

シーン3: 新機能の提案をする時

準備するもの:

  • ユーザーストーリー
  • 画面設計書（ワイヤーフレーム）
  • 類似サービスの参考事例
  • 想定工数とスケジュール

理解度チェック問題 6-1

  1. サイトの表示速度を測定できるGoogleツールは？
  2. バグレポートに必須の情報を3つ挙げてください
  3. 予算削減の提案として適切でないものは？ a)機能削減 b)品質基準下げ c)段階リリース
  4. CVR改善でまず確認すべき項目は？

解答: 1. PageSpeed Insights 2. 再現手順・発生環境・期待動作（など） 3. b)品質基準下げ 4. ユーザーの離脱ポイント

7. AI活用実践編

🎯 なぜ必要か

業務スピードと質を同時に向上させ、世界中の同世代に対して圧倒的な競争優位を築くため。

7.1 プロンプトエンジニアリング基礎

良いプロンプトの4要素

  1. 役割設定: 「あなたは〇〇の専門家です」
  2. 具体的指示: 何をしてほしいかを明確に
  3. 制約条件: 文字数、形式、注意点
  4. 出力形式: どのような形で回答してほしいか

悪いプロンプト例

資料作って

良いプロンプト例

あなたは経験豊富な営業資料作成の専門家です。
スポーツメディア向けの広告提案資料を作成してください。

【条件】
- クライアント: 大手スポーツ用品メーカー
- 商品: 新作ランニングシューズ  
- ターゲット: 20-30代ランニング愛好家
- 予算: 月500万円
- 期間: 3ヶ月キャンペーン

【出力形式】
1. 提案概要（100文字以内）
2. ターゲット分析
3. 配信プラン
4. 期待効果
5. 予算配分

### 9.2 実践問題（ケーススタディ）

#### ケース1: 新規サイト制作
**状況**: 
スポーツ用品ECサイトの制作を担当。クライアントから以下の相談：
- 「予算300万円でスマホ対応サイトを作りたい」
- 「3ヶ月後にはオープンしたい」  
- 「SNS連携もしたい」

**問題**:
1. 技術的制約で確認すべき点を3つ挙げる
2. 予算内で実現可能な提案を考える
3. スケジュールの妥当性を判断する

#### ケース2: 広告効果改善
**状況**:
動画広告キャンペーンの効果が悪い
- インプレッション: 100万
- クリック: 500
- コンバージョン: 10
- 広告費: 50万円

**問題**:
1. CTRとCVRを計算する
2. 問題点を特定する
3. 改善提案を3つ考える

#### ケーススタディ解答例

**ケース1解答例**:
1. 技術的確認点: 決済システム、在庫管理連携、セキュリティ要件
2. 提案: 既存ECプラットフォーム活用、段階的機能追加
3. スケジュール: 300万円規模なら可能だが、要件次第

**ケース2解答例**:
1. CTR: 0.05%, CVR: 2%  
2. 問題: CTRが低すぎる（業界平均0.1-0.3%）
3. 改善: 動画クリエイティブ見直し、ターゲティング精緻化、配信面見直し
