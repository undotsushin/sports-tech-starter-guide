## 2.1 ブラウザの基礎知識

### ブラウザとは

**簡単に言うと**: インターネットでWebサイトを見るためのアプリ

**定義**:
- Webページを表示するためのソフトウェア
- スマホにもPCにも必ず入っている
- 普段何気なく使っているアプリです

**主要ブラウザ**:
1. **Google Chrome** (クローム)
   - 世界シェアNo.1
   - Googleが開発
   - 拡張機能が豊富

2. **Safari** (サファリ)
   - iPhoneやMacに標準搭載
   - Apple製品との相性が良い
   - iOS (iPhone/iPad) ではデフォルトブラウザ

3. **Microsoft Edge** (エッジ)
   - Windowsに標準搭載
   - 以前のInternet Explorerの後継

4. **Firefox** (ファイアフォックス)
   - オープンソースで開発
   - プライバシー保護に強い

**実務での重要性**:
ブラウザによって見た目や動作が微妙に異なることがあります。そのため「どのブラウザまで対応するか」を最初に決める必要があります。

### ブラウザ対応の考え方

**簡単に言うと**: 古いブラウザにも対応するか、新しいブラウザだけ対応するかを決める

**モダンブラウザ対応**:
- 最新版や比較的新しいバージョンのブラウザのみサポート
- 新しい技術を使えるので、開発が楽で機能も豊富
- 開発コストが低い
- 例: Chrome最新版、Safari最新版、Edge最新版

**レガシーブラウザ対応**:
- 古いブラウザ（Internet Explorer 11など）にも対応
- 古い技術しか使えないので、開発が大変
- 開発コストが高い
- 現在は対応しないケースが増えている

**実務での判断基準**:

```
ターゲットユーザーの分析
        ↓
┌──────────────────────────┐
│ 一般消費者向けサービス？   │ → モダンブラウザのみ対応
│ (EC、メディア、SNSなど)   │   (コスト削減、最新機能)
└──────────────────────────┘

┌──────────────────────────┐
│ 企業・官公庁向けサービス？ │ → レガシー対応が必要な場合も
│ (BtoB、行政システムなど)  │   (古いPC環境が残っている)
└──────────────────────────┘
```

**実務での会話例**:
- 「このサイトはIE11対応が必要ですか？対応すると工数が1.5倍になります」
- 「ユーザーの95%がモダンブラウザなので、IE対応は切りましょう」
- 「Safariで表示崩れがあるので、iOS実機で確認お願いします」

**よくある質問**:
- Q: 全てのブラウザで完璧に同じ表示にできますか？
- A: 難しいです。各ブラウザで微妙な違いがあります。ただし「見た目が多少違っても、機能は使える」状態を目指すのが一般的です。

## 2.2 ドメインとURL

### ドメイン（Domain）

**簡単に言うと**: インターネット上の「住所」や「店の名前」のようなもの

**定義**:
- Webサイトの場所を示す文字列
- 世界で1つだけのユニークな名前
- 年間数千円〜数万円で取得・維持できる

**具体例**:
- `sportsbull.jp` (運動通信社のドメイン)
- `google.com` (Googleのドメイン)
- `amazon.co.jp` (Amazon日本のドメイン)

**ドメインの種類と構造**:

```
        サブドメイン  セカンドレベル  トップレベル
              ↓          ↓           ↓
        blog.sportsbull.jp
        ────  ──────────  ──
         │        │        └── .jp (日本)
         │        │            .com (商用)
         │        │            .org (組織)
         │        └── 企業やサービス名
         └── 用途別に分けた部分
```

**トップレベルドメイン（TLD）の種類**:
- **.jp**: 日本を示す（日本の組織のみ取得可能）
- **.com**: 商用サイト（世界中で取得可能）
- **.co.jp**: 日本の企業（登記された企業のみ）
- **.ne.jp**: 日本のネットワークサービス
- **.org**: 非営利組織
- **.edu**: 教育機関

**サブドメインの使い方**:
- `blog.sportsbull.jp` - ブログ用
- `shop.sportsbull.jp` - ショップ用
- `api.sportsbull.jp` - API用
- 1つのドメインから複数のサブドメインを無料で作れる

**実務での会話例**:
- 「ドメインは .jp と .com どちらにしますか？」
- 「サブドメインでブログを分けた方が管理しやすいです」
- 「このドメインは既に取得されているので、別の名前を考えましょう」

**よくある質問**:
- Q: ドメインはいくらで取得できますか？
- A: 年間1,000円〜3,000円程度が一般的。.co.jpなど信頼性の高いものは年間数千円〜1万円程度です。

### URL（Uniform Resource Locator）

**簡単に言うと**: Webページの「完全な住所」

**定義**:
- インターネット上のページの正確な場所を示す文字列
- ブラウザのアドレスバーに表示されるもの
- リンクとして使える

**URLの構造**:

```
https://www.sportsbull.jp/news/article/123?category=soccer#title
─┬──  ───┬──────────── ─┬─────────────  ─┬──────────── ──┬──
 │       │               │                 │               │
プロトコル ドメイン      パス           クエリパラメータ  フラグメント
```

**各要素の説明**:

1. **プロトコル** (`https://`)
   - 通信方法を指定
   - `https://`: 暗号化された安全な通信（推奨）
   - `http://`: 暗号化されていない通信（非推奨）

2. **ドメイン** (`www.sportsbull.jp`)
   - サイトの住所
   - `www` はサブドメインの一種（省略されることも多い）

3. **パス** (`/news/article/123`)
   - サイト内のどのページかを示す
   - フォルダ構造のような階層構造
   - 例: `/news/` = ニュース一覧、`/news/article/123` = 記事No.123

4. **クエリパラメータ** (`?category=soccer`)
   - ページに渡す追加情報
   - `?` の後に `キー=値` の形式で指定
   - 複数ある場合は `&` で繋ぐ
   - 例: `?category=soccer&year=2025`

5. **フラグメント** (`#title`)
   - ページ内の特定の場所にジャンプ
   - `#` の後に指定
   - 例: `#comments` でコメント欄にスクロール

**実務での会話例**:
- 「このページのURLを教えてください」
- 「URLパラメータで検索条件を渡します」
- 「URLが長すぎるとSEOに悪影響があります」
- 「URLは日本語ではなく英数字にしてください」

**よくある失敗例**:
- URLに日本語を使うと文字化けする
- URLが変わるとSEOの評価がリセットされる
- スペースや特殊文字を使うとエラーになる

**URLの良い例と悪い例**:

良い例:
```
https://example.com/news/2025/sports-tech-guide
→ 短く、意味がわかりやすい
```

悪い例:
```
https://example.com/page.php?id=12345&category=999
→ 数字だけで何のページかわからない
```

## 2.3 サーバーとホスティング

### サーバー（Server）

**簡単に言うと**: Webサイトのデータを保存して、見たい人に届けるコンピュータ

**役割**:
- Webサイトのファイル（HTML、画像、動画など）を保存
- ユーザーがアクセスしたときにデータを送信
- 24時間365日稼働し続ける

**サーバーの仕組み**:

```
     ユーザー                サーバー
    (あなた)               (データ保管庫)
        │                      │
        │  1. ページを見たい!   │
        ├──────────────────────>│
        │                      │
        │                      │ 2. データを探す
        │                      │    ↓
        │                      │   [HTML]
        │                      │   [画像]
        │                      │   [CSS]
        │                      │
        │  3. はい、どうぞ！    │
        │<──────────────────────┤
        │                      │
    ページ表示
```

**サーバーの種類**:

1. **共有サーバー（レンタルサーバー）**
   - 費用: 月額500円〜2,000円程度
   - 特徴: 1台のサーバーを複数のサイトで共用
   - メリット: 安い、初心者でも簡単
   - デメリット: 他のサイトの影響を受けることがある
   - 向いているサイト: 個人ブログ、小規模企業サイト

2. **専用サーバー（VPS）**
   - 費用: 月額数千円〜数万円
   - 特徴: 1つのサイト専用のサーバー
   - メリット: 高速、自由にカスタマイズ可能
   - デメリット: 高い、技術知識が必要
   - 向いているサイト: 中〜大規模サイト、アクセスの多いサイト

3. **クラウドサーバー**
   - 費用: 使った分だけ課金（従量課金）
   - 代表的なサービス: AWS、Google Cloud、Azure
   - メリット: 柔軟にスペック変更可能、世界中にデータセンター
   - デメリット: 複雑、使いすぎると高額になる
   - 向いているサイト: 大規模サイト、アクセス変動が大きいサイト

**実務での会話例**:
- 「月間PV10万程度なら共有サーバーで十分です」
- 「アクセスが集中する可能性があるので、クラウドサーバーにしましょう」
- 「サーバーのスペックを上げないと、表示が遅くなります」

**よくある質問**:
- Q: サーバーはどこで借りられますか？
- A: 国内だとさくらインターネット、エックスサーバー、ConoHaなどが有名です。

### SSL証明書

**簡単に言うと**: サイトの通信を暗号化して安全にする「鍵」

**役割**:
1. **データの暗号化**: 送受信するデータを第三者に見られないようにする
2. **サイトの信頼性証明**: このサイトが本物であることを証明
3. **SEO効果**: Googleが SSL対応サイトを優遇

**SSL対応の見分け方**:

```
SSL対応あり（安全）
┌────────────────────────────┐
│  https://example.com      │  ← 「https」で始まる
│  🔒 保護された通信         │  ← 鍵マークが表示
└────────────────────────────┘

SSL対応なし（危険）
┌────────────────────────────┐
│  http://example.com       │  ← 「http」で始まる
│  ⚠ 保護されていません      │  ← 警告が表示される
└────────────────────────────┘
```

**SSL証明書の種類**:

1. **ドメイン認証（DV）**
   - 費用: 無料〜年間数千円
   - 取得時間: 数分〜数時間
   - 信頼度: 低
   - 用途: 個人サイト、小規模サイト

2. **企業認証（OV）**
   - 費用: 年間数万円
   - 取得時間: 数日
   - 信頼度: 中
   - 用途: 企業サイト

3. **EV認証（Extended Validation）**
   - 費用: 年間10万円以上
   - 取得時間: 数週間
   - 信頼度: 高（アドレスバーに企業名が表示される）
   - 用途: ECサイト、金融機関

**実務での重要性**:
- **SEO**: GoogleはSSL対応サイトを検索順位で優遇
- **ユーザーの信頼**: SSL非対応だと「危険なサイト」と警告が出る
- **必須**: 現代ではSSL対応は必須。非対応だとユーザーが離脱する

**実務での会話例**:
- 「SSL証明書の期限が切れています。すぐ更新してください」
- 「無料のLet's Encryptで十分です」
- 「フォームがあるサイトなので、必ずSSL対応してください」

## 2.4 レスポンシブデザイン

**簡単に言うと**: スマホでもPCでも見やすく表示される仕組み

**定義**:
- 1つのWebサイトが、画面サイズに応じて自動的に最適な表示に変わる技術
- スマホ、タブレット、PC、それぞれの画面で見やすくなる
- 現代のWebサイトでは必須の技術

**レスポンシブデザインのイメージ**:

```
PC表示（横長）
┌─────────────────────────────────────┐
│ ヘッダー  [メニュー][メニュー][検索] │
├──────────┬──────────────────────────┤
│サイドバー│    メインコンテンツ       │
│ ・記事  │    記事本文が            │
│ ・広告  │    ここに表示される       │
└──────────┴──────────────────────────┘

スマホ表示（縦長）
┌─────────────┐
│   ヘッダー   │
│     ≡       │ ← ハンバーガーメニュー
├─────────────┤
│メインコンテンツ│
│  記事本文   │
│  が縦に    │
│  並ぶ      │
├─────────────┤
│サイドバー   │
│  ・記事    │ ← 下に移動
│  ・広告    │
└─────────────┘
```

**必要性**:
- 日本のWebサイトアクセスの約70%がスマホから
- スマホ対応していないと、Googleの検索順位が下がる
- ユーザーがすぐに離脱してしまう

**実装方法**:
- CSSの「メディアクエリ」という機能を使用
- 画面幅に応じてレイアウトを切り替える
- デザイナーとエンジニアの両方の理解が必要

**実務での会話例**:
- 「スマホでメニューが見切れているので、ハンバーガーメニューにしてください」
- 「このサイトはレスポンシブ対応していますか？」
- 「実機で確認したら、ボタンが小さすぎてタップできませんでした」

**よくある失敗例**:
- PCだけで確認して、スマホで崩れている
- ボタンが小さすぎてスマホでタップできない
- 横スクロールが発生してしまう
- 文字が小さすぎて読めない

**チェックポイント**:
1. 実機（実際のスマホ）で必ず確認する
2. iPhone と Android の両方で確認
3. 縦向き・横向き両方で確認
4. 文字サイズは最低14px以上
5. ボタンのタップ領域は最低44px四方
