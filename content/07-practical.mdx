## 6.1 クライアント対応シーン

実際の業務でよくあるクライアントからの相談と、その対応方法を学びましょう。

### シーン1: 「サイトが重い」と言われた時

**簡単に言うと**: サイトの表示が遅いという相談への対応方法

**よくあるやり取り**:
```
クライアント: 「うちのサイト、最近すごく重いんですけど...」
あなた: 「どのページが、どのくらい遅いか教えていただけますか？」
クライアント: 「トップページで、5秒くらいかかります」
```

**よくある原因ベスト5**:

1. **画像サイズが大きすぎる（70%）**
   - 5MB以上の写真をそのままアップロード
   - 画像圧縮をしていない
   - 必要以上に高解像度

2. **サーバーのスペック不足（15%）**
   - アクセス数に対してサーバーが弱い
   - 共用サーバーで他のサイトの影響を受けている
   - メモリ不足

3. **プラグインが多すぎる（10%）**
   - 使っていないプラグインが残っている
   - 重いプラグインを複数使用
   - プラグイン同士の競合

4. **データベースの肥大化（3%）**
   - 古いデータが溜まっている
   - 最適化されていない

5. **外部スクリプトの読み込みが遅い（2%）**
   - 広告タグが多すぎる
   - SNSボタンが複数

**対応フロー**:

```
STEP 1: ヒアリング
┌─────────────────────────────┐
│いつから？                    │
│どのページが？               │
│どのくらい遅い？（秒数）      │
│スマホ？PC？                 │
└─────────────────────────────┘
          ↓
STEP 2: 測定
┌─────────────────────────────┐
│PageSpeed Insights で測定     │
│スコアを確認（100点満点）     │
│改善提案を確認               │
└─────────────────────────────┘
          ↓
STEP 3: 原因特定
┌─────────────────────────────┐
│画像サイズをチェック          │
│サーバーログを確認           │
│プラグイン一覧を確認         │
└─────────────────────────────┘
          ↓
STEP 4: 改善提案
┌─────────────────────────────┐
│具体的な改善策を提示         │
│費用と効果を説明             │
│優先順位をつける             │
└─────────────────────────────┘
```

**改善提案の具体例**:

1. **画像最適化（費用: 0円〜5万円）**
   - 「すべての画像を圧縮すれば、50%速くなります」
   - 「WebP形式に変換しましょう」
   - 作業時間: 1〜3日

2. **サーバー強化（費用: 月額+5,000円〜）**
   - 「現在のサーバーはアクセス数に対して弱いです」
   - 「上位プランにすれば、読み込みが2倍速くなります」
   - 切り替え時間: 即日

3. **不要プラグイン削除（費用: 0円〜3万円）**
   - 「使っていないプラグインが10個あります」
   - 「これらを削除すれば30%改善します」
   - 作業時間: 1日

**実務での会話例**:

```
クライアント: 「サイトが重いんですが、直せますか？」

悪い回答:
「画像が重いんじゃないですかね。圧縮すればいいと思います」
→ 測定もせず憶測で答えている

良い回答:
「まずPageSpeed Insightsで測定させてください。
原因を特定してから、費用と効果をご提案します。
今日測定して、明日には改善案をお送りできます」
→ 手順を明確にし、期限も提示している
```

**よくある質問**:
- Q: PageSpeed Insightsのスコアは何点あればいいですか？
- A: 90点以上が理想ですが、70点以上あれば実用上問題ありません。50点以下は改善が必要です。

### シーン2: 「CVRが悪い」と相談された時

**簡単に言うと**: コンバージョン率（購入率）が低いという相談への対応

**定義**:
- CVR（Conversion Rate）= コンバージョン率
- サイトに来た人のうち、何%が購入や申込をしたか
- 例: 100人来て3人が購入 → CVR 3%

**よくあるやり取り**:
```
クライアント: 「CVRが悪いんです。どうにかなりませんか？」
あなた: 「現在のCVRは何%ですか？」
クライアント: 「0.5%くらいです」
あなた: 「業界平均が1〜3%なので、確かに低いですね。
        どこで離脱しているか分析しましょう」
```

**確認ポイントチェックリスト**:

```
□ 現在のCVR数値
□ 業界平均との比較
   - ECサイト: 1〜3%
   - BtoB問い合わせ: 2〜5%
   - アプリダウンロード: 10〜30%
□ 月間の訪問者数
□ どのページで離脱しているか
□ 平均滞在時間
□ モバイル/PCの比率
```

**離脱ポイントの分析**:

```
100人がサイトに来た場合
    ↓
┌─────────────────┐
│トップページ     │ 100人
└─────────────────┘
    ↓ 30人離脱（興味なし）
┌─────────────────┐
│商品ページ       │ 70人
└─────────────────┘
    ↓ 40人離脱（価格が高い？情報不足？）
┌─────────────────┐
│カート追加       │ 30人
└─────────────────┘
    ↓ 20人離脱（送料が高い？入力面倒？）
┌─────────────────┐
│購入完了         │ 10人（CVR 10%）
└─────────────────┘

→ どこで一番離脱しているかを特定
```

**改善提案の優先順位**:

**【優先度 高】すぐできて効果大**
1. **CTAボタンの改善（費用: 0円〜5万円）**
   - 色を変える: 緑→オレンジで20%向上
   - 位置を変える: 下部→上部に追加
   - 文言を変える: 「購入」→「今すぐ購入して○○を手に入れる」
   - 作業時間: 1日

2. **フォームの項目数削減（費用: 3万円〜10万円）**
   - 15項目→5項目に削減でCVR 2倍になった事例多数
   - 必須項目を減らす
   - 住所は郵便番号から自動入力
   - 作業時間: 3〜5日

**【優先度 中】時間はかかるが効果あり**
3. **ページ表示速度の改善（費用: 5万円〜30万円）**
   - 3秒→1秒でCVR 25%向上
   - 画像圧縮、サーバー強化
   - 作業時間: 1〜2週間

4. **モバイル対応の見直し（費用: 10万円〜50万円）**
   - モバイルユーザーの70%が離脱している場合
   - レスポンシブデザインの改善
   - 作業時間: 2〜4週間

**【優先度 低】効果は限定的**
5. **デザインの全面刷新（費用: 50万円〜200万円）**
   - 見た目は良くなるが、CVRは10%程度しか上がらない
   - 他の改善をすべてやってから

**実務での会話例**:

```
クライアント: 「CVRを3%にしたいです」

悪い回答:
「デザインを変えましょう。100万円かかりますが」
→ 高額な提案を最初にしている

良い回答:
「まず、フォーム項目を15個→5個に減らしましょう。
費用5万円、1週間で対応でき、CVRが1.5倍になる可能性があります。
それでも目標に届かなければ、次の施策を提案します」
→ 低コスト・短期間・効果的な提案から
```

**A/Bテストの提案**:

```
現在のページ（A）と改善版（B）を比較テスト

例:
┌─────────────┬─────────────┐
│ A版         │ B版         │
├─────────────┼─────────────┤
│緑色ボタン    │オレンジボタン│
│「購入」      │「今すぐ試す」│
├─────────────┼─────────────┤
│CVR 1.2%    │CVR 1.8%     │
└─────────────┴─────────────┘
→ B版が50%も良い！B版を採用
```

**よくある質問**:
- Q: CVR改善にどのくらい時間がかかりますか？
- A: 簡単な改善（ボタン変更など）は1週間、大規模改善（LP全面刷新など）は1〜3ヶ月です。段階的に改善していくのがおすすめです。

### シーン3: 「予算を抑えたい」と言われた時

**簡単に言うと**: コストを削減しながら目的を達成する提案

**よくあるやり取り**:
```
クライアント: 「見積もり300万円ですか...もっと安くなりませんか？」
あなた: 「承知しました。どの機能が最も重要ですか？」
クライアント: 「とにかく商品販売ができればいいです」
あなた: 「では、まずは最小限の機能で始めて、
        売上が出てから拡張しましょう」
```

**予算削減の4つのアプローチ**:

```
元の見積もり: 300万円
    ↓
┌──────────────────────────────┐
│1. 機能の優先順位付け（MVP開発）│
│   → 200万円（33%削減）          │
└──────────────────────────────┘
    ↓
┌──────────────────────────────┐
│2. 既存テンプレートの活用       │
│   → 150万円（50%削減）          │
└──────────────────────────────┘
    ↓
┌──────────────────────────────┐
│3. 段階的リリース               │
│   → 第1期100万円＋第2期100万円 │
└──────────────────────────────┘
    ↓
┌──────────────────────────────┐
│4. オープンソースの活用         │
│   → 80万円（73%削減）           │
└──────────────────────────────┘
```

**1. 機能の優先順位付け（MVP開発）**

**簡単に言うと**: 最小限の機能で始めて、後から追加していく

**MVP = Minimum Viable Product（実用最小限の製品）**

**例: ECサイトの場合**
```
フル機能版（300万円）    MVP版（200万円）
┌────────────────┐    ┌────────────────┐
│商品一覧 ○      │    │商品一覧 ○      │
│商品詳細 ○      │    │商品詳細 ○      │
│カート ○        │    │カート ○        │
│決済 ○          │    │決済 ○          │
│会員登録 ○      │    │会員登録 ○      │
├────────────────┤    ├────────────────┤
│レビュー機能 ○  │    │レビュー機能 ×  │← 後で追加
│クーポン機能 ○  │    │クーポン機能 ×  │← 後で追加
│ポイント機能 ○  │    │ポイント機能 ×  │← 後で追加
│おすすめ機能 ○  │    │おすすめ機能 ×  │← 後で追加
│ランキング ○    │    │ランキング ×    │← 後で追加
└────────────────┘    └────────────────┘
```

**メリット**:
- 初期費用が33%削減
- 早くリリースできる（3ヶ月→2ヶ月）
- 実際に使ってから必要な機能を追加できる

**デメリット**:
- 後から追加する費用がかかる
- 競合より機能が少ない

**2. 既存テンプレートの活用**

**簡単に言うと**: ゼロから作らず、既存のデザインテンプレートを使う

**フルスクラッチ vs テンプレート**
```
フルスクラッチ              テンプレート活用
（完全オリジナル）          （既存デザイン利用）

費用: 300万円               費用: 150万円（50%削減）
期間: 4ヶ月                 期間: 2ヶ月（半分）
デザイン: 完全自由          デザイン: ある程度制限
独自性: 高い                独自性: 低い
```

**代表的なテンプレート**:
- WordPress: 無料〜3万円（テーマ代）
- Shopify: 無料〜3万円（テーマ代）+ 月額3,000円
- Wix: 月額1,000円〜3,000円

**実務での会話例**:
```
クライアント: 「オリジナルデザインがいいです」
あなた: 「承知しました。ただ、まずはテンプレートで始めて、
        売上が月100万円を超えたらオリジナルに作り変える
        という方法もあります。初期費用を150万円削減できます」
```

**3. 段階的リリース**

**簡単に言うと**: 一度に作らず、何回かに分けてリリース

```
一括リリース              段階的リリース

第1期: 全機能開発          第1期: 基本機能のみ
期間: 4ヶ月                期間: 2ヶ月
費用: 300万円              費用: 100万円
リリース: 4ヶ月後          リリース: 2ヶ月後
                               ↓
                          第2期: 追加機能開発
                          期間: 2ヶ月
                          費用: 100万円
                               ↓
                          第3期: 拡張機能開発
                          期間: 2ヶ月
                          費用: 100万円
```

**メリット**:
- 初期投資が少ない（100万円）
- 早くリリースできる（2ヶ月）
- ユーザーの反応を見ながら開発できる
- 途中で方向転換できる

**デメリット**:
- トータルコストは同じか、少し高くなる
- 何度も調整が必要

**4. オープンソースの活用**

**簡単に言うと**: 無料のソフトウェアを使って開発費を削減

**オープンソース = 無料で使えるソフトウェア**

```
フルスクラッチ              オープンソース活用

ECサイト構築: 300万円       WordPress + WooCommerce
カスタマイズ開発            カスタマイズのみ
すべて一から作る            基本機能は無料
                          費用: 80万円（73%削減）
```

**代表的なオープンソース**:
- WordPress: Webサイト、ブログ（無料）
- WooCommerce: ECサイト（無料）
- Moodle: eラーニングシステム（無料）
- Concrete5: 企業サイト（無料）

**メリット**:
- 初期費用が大幅削減（50〜80%）
- セキュリティアップデートが自動
- プラグインが豊富

**デメリット**:
- デザインの自由度が低い
- 独自機能は別途開発が必要
- サーバー費用は別途必要（月1万円程度）

**実務での会話例**:

```
クライアント: 「予算が200万円しかありません」

悪い回答:
「それだと何もできません。最低300万円必要です」
→ 選択肢を提示していない

良い回答:
「200万円でしたら、3つの選択肢があります。
1. MVP開発: 基本機能のみで200万円
2. テンプレート活用: フル機能で150万円
3. 段階的リリース: 第1期100万円＋第2期100万円

それぞれメリット・デメリットをご説明しますので、
ご希望に合うものを選んでいただけますか？」
→ 複数の選択肢を提示し、判断を委ねている
```

**やってはいけない提案**:
- × 品質基準を下げる（バグが多くても良しとする）
- × セキュリティ対策を省く（個人情報漏洩のリスク）
- × テストを省略する（不具合だらけでリリース）
- × 保守サポートなし（トラブル時に対応できない）

**よくある質問**:
- Q: どの方法が一番おすすめですか？
- A: 初めてのサイト構築なら「テンプレート活用」が最もコスパが良いです。独自性が必要なら「MVP開発」、キャッシュフローが厳しいなら「段階的リリース」がおすすめです。

## 6.2 社内エンジニア連携シーン

エンジニアとのコミュニケーションは、プロジェクトの成功を左右します。適切な依頼方法を学びましょう。

### シーン1: 仕様変更を依頼する時

**簡単に言うと**: エンジニアに機能の変更をお願いする時の正しい伝え方

**よくある失敗例と改善例**:

```
【悪い例】
「ボタンの色を青から赤に変えてください。
簡単だと思うので、明日までにお願いします」

問題点:
- 「簡単だと思うので」→ エンジニアの工数を勝手に決めつけている
- 明日まで → 期限が短すぎる
- 理由がない → なぜ変更するのか不明

【良い例】
「A/Bテストの結果、赤色ボタンのCVRが20%高かったため、
メインボタンの色を青から赤に変更したいです。
工数はどのくらいかかりますか？
急ぎではないので、他のタスクの合間で対応いただけますか？」

良い点:
- 変更理由が明確（A/Bテストの結果）
- 工数見積もりを依頼している
- 優先度を相談している
```

**やってはいけないこと ワースト5**:

1. **「簡単だと思うので」という前置き**
   - エンジニアから最も嫌がられる言葉No.1
   - 工数は実際にやる人が決めるもの
   - 「簡単だと思うなら自分でやって」と思われる

2. **期限直前での依頼**
   - 「明日リリースなので今日中にお願いします」
   - エンジニアには他のタスクもある
   - 最低でも1週間前には依頼

3. **曖昧な指示**
   - 「いい感じにお願いします」
   - 「適当に」「よしなに」
   - 具体的な要件を文書で

4. **口頭だけで済ませる**
   - 「さっき言ったやつ、やってね」
   - 言った言わないのトラブルになる
   - 必ず文書（メール、チャット）で残す

5. **何度も仕様を変える**
   - 「やっぱりこっちがいい」を繰り返す
   - エンジニアの時間を無駄にしている
   - 最初に要件を固める

**正しいアプローチ（5ステップ）**:

```
STEP 1: 変更理由を明確に説明
┌─────────────────────────────┐
│なぜ変更するのか？           │
│データや根拠はあるか？       │
│変更しないとどうなるか？     │
└─────────────────────────────┘
          ↓
STEP 2: 具体的な要件を文書で共有
┌─────────────────────────────┐
│何を変更するのか？           │
│どう変更するのか？           │
│スクリーンショットや図を添付 │
└─────────────────────────────┘
          ↓
STEP 3: 工数見積もりを依頼
┌─────────────────────────────┐
│「工数はどのくらいですか？」 │
│「いつ頃着手できますか？」   │
└─────────────────────────────┘
          ↓
STEP 4: 優先度を相談
┌─────────────────────────────┐
│急ぎ度: 高/中/低             │
│他のタスクとの兼ね合い       │
│リリース希望日               │
└─────────────────────────────┘
          ↓
STEP 5: 完了確認
┌─────────────────────────────┐
│テストして動作確認           │
│お礼を伝える                 │
│フィードバックを共有         │
└─────────────────────────────┘
```

**依頼文のテンプレート**:

```
件名: 【仕様変更依頼】ログインボタンの色変更

◆変更理由
A/Bテストの結果、赤色ボタンのCVRが青色より20%高いことが
判明したため、ユーザーの利便性向上のため変更したいです。

◆変更内容
- 対象: トップページのログインボタン
- 変更前: 青色（#0000FF）
- 変更後: 赤色（#FF0000）
- 添付: 修正イメージ図（image.png）

◆希望納期
リリース希望: 2週間後（11月10日）
※急ぎではないので、他のタスクの優先度次第で調整可能です

◆その他
工数見積もりと着手可能時期を教えていただけますか？
よろしくお願いします。
```

**実務での会話例**:

```
あなた: 「お疲れ様です。仕様変更の相談があるのですが、
        今お時間よろしいですか？」
エンジニア: 「はい、大丈夫です」

        A/Bテストで赤色の方がCVRが20%高かったので...」
```

### シーン2: バグを発見した時

**簡単に言うと**: 不具合を見つけた時の正しい報告方法

**バグとは**:
- システムが期待通りに動かないこと
- 例: ボタンを押しても何も起きない、エラーが表示される

**よくある失敗例と改善例**:

```
【悪い例】
「ログインできません。直してください」

問題点:
- どこで？いつ？どうやって？が不明
- エンジニアが再現できない
- 修正に時間がかかる

【良い例】
【発生環境】
- OS: Windows 11
- ブラウザ: Chrome 120
- デバイス: PC

【再現手順】
1. トップページにアクセス
2. 「ログイン」ボタンをクリック
3. メールアドレス「test@example.com」を入力
4. パスワード「12345678」を入力
5. 「ログイン」ボタンをクリック

【期待する動作】
マイページに遷移する

【実際の動作】
「エラーが発生しました」と表示され、ログインできない

【添付】
スクリーンショット（error.png）
```

**バグレポートに必須の5項目**:

```
1. 発生環境
┌─────────────────────────────┐
│□ OS（Windows/Mac/iOS/Android）│
│□ ブラウザ（Chrome/Safari/等） │
│□ デバイス（PC/スマホ/タブレット）│
│□ 画面サイズ                   │
└─────────────────────────────┘

2. 再現手順
┌─────────────────────────────┐
│1. ○○にアクセス               │
│2. △△をクリック               │
│3. □□を入力                   │
│ → 誰でも同じ結果になる手順   │
└─────────────────────────────┘

3. 期待する動作
┌─────────────────────────────┐
│本来はこうなるはず             │
│正常な動作の説明               │
└─────────────────────────────┘

4. 実際の動作
┌─────────────────────────────┐
│実際にはこうなった             │
│エラーメッセージの全文         │
└─────────────────────────────┘

5. スクリーンショットまたは動画
┌─────────────────────────────┐
│画像: 問題が起きている画面     │
│動画: 操作の様子（loom等）     │
└─────────────────────────────┘
```

**バグの優先度の付け方**:

```
【優先度: 高】すぐに修正が必要
┌─────────────────────────────┐
│- サービスが使えない           │
│- 個人情報が漏れる可能性       │
│- 決済ができない               │
│- 全ユーザーに影響             │
└─────────────────────────────┘
→  「至急対応お願いします」

【優先度: 中】できるだけ早く修正
┌─────────────────────────────┐
│- 一部の機能が使えない         │
│- 特定の環境でのみ発生         │
│- ユーザー体験が悪化           │
└─────────────────────────────┘
→ 「1週間以内に対応お願いします」

【優先度: 低】時間のある時に修正
┌─────────────────────────────┐
│- 見た目の問題（文字のズレ等） │
│- 稀にしか発生しない           │
│- 回避策がある                 │
└─────────────────────────────┘
→ 「次のアップデートで対応お願いします」
```

**バグ報告のテンプレート**:

```
件名: 【バグ報告】ログインができない（優先度: 高）

◆発生環境
- OS: Windows 11
- ブラウザ: Chrome 120.0
- デバイス: PC（デスクトップ）

◆再現手順
1. https://example.com にアクセス
2. 右上の「ログイン」ボタンをクリック
3. メールアドレス「test@example.com」を入力
4. パスワード「12345678」を入力
5. 「ログイン」ボタンをクリック

◆期待する動作
マイページ（/mypage）に遷移する

◆実際の動作
「エラーが発生しました。もう一度お試しください」
というメッセージが表示され、ログインできない

◆その他
- Safari では正常にログインできる
- 他のユーザーも同じ問題が発生している模様
- 添付: スクリーンショット（error.png）

◆優先度
高（ユーザーがログインできない状態）

よろしくお願いします。
```

**実務での会話例**:

```
あなた: 「お疲れ様です。バグを見つけたので報告します」
エンジニア: 「ありがとうございます。どんなバグですか？」
あなた: 「Chromeでログインができません。
        詳細はSlackに書いたので確認お願いします」
エンジニア: 「了解です。Safariでは動きますか？」
あなた: 「はい、Safariは正常です」
エンジニア: 「Chrome固有の問題ですね。調査します」
```

**よくある質問**:
- Q: バグかどうかわからない時はどうすればいいですか？
- A: 「これは仕様ですか？それともバグですか？」と聞いてください。仕様書と動作が違う場合は報告した方が良いです。

### シーン3: 新機能の提案をする時

**簡単に言うと**: 新しい機能を作りたい時の提案方法

**よくある失敗例と改善例**:

```
【悪い例】
「Instagramみたいな機能を追加してください」

問題点:
- 「Instagramみたいな」が曖昧
- 写真投稿？いいね？コメント？
- 工数が全く見えない

【良い例】
「ユーザーが写真を投稿できる機能を追加したいです。
参考: Instagram の投稿機能
理由: ユーザーから「試合の写真を共有したい」という要望が月20件
想定工数: 2〜3週間？
添付: 画面設計書（wireframe.pdf）」
```

**準備するもの（4点セット）**:

```
1. ユーザーストーリー
┌─────────────────────────────┐
│誰が？                         │
│何をしたい？                   │
│なぜ？                         │
└─────────────────────────────┘
例:
「スポーツチームのコーチが、
試合の動画を選手と共有したい。
なぜなら、遠征中の選手も確認できるようにするため」

2. 画面設計書（ワイヤーフレーム）
┌─────────────────────────────┐
│画面のレイアウト               │
│ボタンの配置                   │
│入力項目                       │
└─────────────────────────────┘
ツール: Figma、Adobe XD、手書きでもOK

3. 類似サービスの参考事例
┌─────────────────────────────┐
│他のサービスでの実装例         │
│スクリーンショット             │
│良い点・悪い点                 │
└─────────────────────────────┘

4. 想定工数とスケジュール
┌─────────────────────────────┐
│ざっくりの工数予想             │
│リリース希望日                 │
│優先度                         │
└─────────────────────────────┘
```

**機能提案フロー**:

```
STEP 1: 背景・目的の説明
┌─────────────────────────────┐
│なぜこの機能が必要？           │
│ユーザーからの要望？           │
│競合にはある？                 │
│数値データはある？             │
└─────────────────────────────┘
          ↓
STEP 2: 具体的な機能説明
┌─────────────────────────────┐
│どんな機能？                   │
│画面設計書を見せる             │
│参考事例を見せる               │
└─────────────────────────────┘
          ↓
STEP 3: 技術的実現性の確認
┌─────────────────────────────┐
│実現可能？                     │
│どのくらいの工数？             │
│技術的な課題は？               │
└─────────────────────────────┘
          ↓
STEP 4: スケジュール調整
┌─────────────────────────────┐
│いつリリースしたい？           │
│他のタスクとの優先度は？       │
│段階的リリースは可能？         │
└─────────────────────────────┘
          ↓
STEP 5: 開発開始
┌─────────────────────────────┐
│要件を文書化                   │
│開発着手                       │
│進捗を定期的に確認             │
└─────────────────────────────┘
```

**機能提案書のテンプレート**:

```
件名: 【新機能提案】動画投稿機能の追加

◆背景・目的
現在、ユーザーから「試合の動画を共有したい」という要望が
月20件寄せられています。競合サービス（○○社）には既に
同様の機能があり、当社も対応する必要があると判断しました。

◆ユーザーストーリー
「スポーツチームのコーチが、試合の動画を選手全員と共有したい。
なぜなら、遠征中の選手も後から確認できるようにするため」

◆機能概要
1. 動画アップロード機能（最大100MB、mp4形式）
2. 動画一覧表示（サムネイル付き）
3. 動画再生機能（ブラウザ内再生）
4. コメント機能（動画に対してコメント可能）

◆画面設計
添付: ワイヤーフレーム（wireframe.pdf）

◆参考事例
- YouTube の動画投稿機能
- Instagram のリール機能
添付: スクリーンショット（reference.png）

◆想定工数・スケジュール
- 想定工数: 3〜4週間？（エンジニア確認必要）
- リリース希望: 2ヶ月後（12月末）
- 優先度: 中（ユーザー要望が多いが、緊急ではない）

◆その他
工数とスケジュールをご確認いただき、実現可能か
ご意見をいただけますでしょうか？
```

**実務での会話例**:

```
あなた: 「新機能の提案があるのですが、お時間よろしいですか？」
エンジニア: 「どんな機能ですか？」
あなた: 「動画投稿機能です。ユーザーから月20件要望が来ています。
        画面設計書を作ったので見ていただけますか？」
エンジニア: 「いいですね。でもサーバー容量が心配です」
あなた: 「100MB制限で、最大1000動画までと考えています」
エンジニア: 「それなら大丈夫そうです。3週間くらいかかりますね」
あなた: 「ありがとうございます！では要件書を作ります」
```

**よくある質問**:
- Q: 機能提案が却下されたらどうすればいいですか？
- A: 却下理由を聞いて、改善案を考えましょう。工数が大きいなら機能を削って小さく始める、技術的に難しいなら代替案を探す、などの対応が可能です。
