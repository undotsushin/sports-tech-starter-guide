## 6.1 パスワード管理

**簡単に言うと**: アカウントを守るための最も基本的で重要な対策

### 危険なパスワード ワースト10

```
絶対に使ってはいけないパスワード

1位: 123456
2位: password
3位: 12345678
4位: qwerty
5位: 123456789
6位: 12345
7位: 1234
8位: 111111
9位: 1234567
10位: dragon

これらは数秒で破られます。
```

### 強いパスワードの条件

**最低限の条件**:
```
× 悪い例: password123
○ 良い例: Kj8$mP2#qL9@nR4!

強いパスワードの条件:
1. 12文字以上
2. 大文字・小文字・数字・記号を含む
3. 辞書にある単語を使わない
4. 個人情報を含めない（誕生日、名前など）
5. サービスごとに違うパスワードを使う
```

**パスワード強度の比較**:
```
パスワード            解読時間        評価
password             1秒以下         × 最悪
password123          1秒以下         × 危険
Password123!         3時間           △ 弱い
P@ssw0rd!2024       3年            ○ 普通
Kj8$mP2#qL9@nR4!    7億年          ◎ 強い
```

### パスワード管理ツール

**簡単に言うと**: すべてのパスワードを安全に保存してくれるツール

**主なパスワード管理ツール**:

1. **1Password**
   - 料金: 月400円
   - 対応: Windows/Mac/iOS/Android
   - 特徴: 使いやすさNo.1
   - 用途: 個人・ビジネス

2. **Bitwarden**
   - 料金: 無料（有料版は月140円）
   - 対応: すべてのOS
   - 特徴: オープンソース
   - 用途: 個人

3. **LastPass**
   - 料金: 無料（有料版は月300円）
   - 対応: すべてのOS
   - 特徴: 有名で実績がある
   - 用途: 個人

**メリット**:
- すべてのパスワードを記憶不要
- 自動で強いパスワードを生成
- 自動入力で楽
- 複数デバイスで同期

**使い方**:
```
STEP 1: パスワード管理ツールをインストール
    ↓
STEP 2: マスターパスワードを設定（1つだけ覚える）
    ↓
STEP 3: サービスごとのパスワードを登録
    ↓
STEP 4: 自動入力で楽々ログイン
```

**実務での会話例**:
- 「パスワード管理ツールを導入して、全員に強いパスワードを使わせましょう」
- 「1Passwordなら、チーム全体でパスワードを共有できます」

### 二要素認証（2FA）

**簡単に言うと**: パスワードに加えて、もう1つの認証を追加する方法

**仕組み**:
```
通常のログイン          二要素認証
┌─────────────┐    ┌─────────────┐
│パスワード入力│    │パスワード入力│
└─────────────┘    └─────────────┘
       ↓                   ↓
┌─────────────┐    ┌─────────────┐
│ログイン完了  │    │スマホに通知  │
└─────────────┘    │6桁の数字入力│
                   └─────────────┘
                          ↓
                   ┌─────────────┐
                   │ログイン完了  │
                   └─────────────┘

パスワードが漏れても、スマホがないとログインできない
```

**二要素認証の種類**:

1. **SMS認証**
   - 電話番号に6桁の数字が届く
   - 手軽だが、SIMスワップ攻撃のリスク
   - セキュリティ: 中

2. **認証アプリ（Google Authenticator等）**
   - アプリに6桁の数字が表示される
   - SMSより安全
   - セキュリティ: 高

3. **物理キー（YubiKey等）**
   - USBキーを挿してログイン
   - 最も安全
   - セキュリティ: 最高

**必ず二要素認証を設定すべきサービス**:
```
□ Gmail（メール）
□ AWS / GCP / Azure（クラウド）
□ GitHub（コード管理）
□ Slack（社内チャット）
□ 銀行・証券口座
□ クレジットカード会社
```

**実務での会話例**:
- 「AWSアカウントには必ず二要素認証を設定してください」
- 「パスワードだけでは不十分です。二要素認証を有効化しましょう」

## 6.2 個人情報保護

**簡単に言うと**: 個人の情報を適切に扱い、漏洩させないこと

### 個人情報とは

**定義**:
- 特定の個人を識別できる情報
- 氏名、住所、電話番号、メールアドレスなど

**個人情報の例**:
```
明らかに個人情報
- 氏名
- 住所
- 電話番号
- メールアドレス
- マイナンバー
- クレジットカード番号
- 銀行口座
- 顔写真

実は個人情報
- IPアドレス（特定個人と紐付く場合）
- Cookie ID（特定個人と紐付く場合）
- 位置情報
- 購買履歴
- 閲覧履歴
```

### 個人情報保護法（日本）

**簡単に言うと**: 日本で個人情報を扱う際のルール

**対象**:
- 日本国内で個人情報を扱うすべての事業者
- 5,000人以上の個人情報を扱う場合、個人情報保護委員会への届出が必要

**違反すると**:
```
罰則
- 法人: 最大1億円の罰金
- 個人: 最大2年の懲役または100万円の罰金
- 行政指導・命令
- 社会的信用の失墜
- 損害賠償請求
```

**守るべき5つのルール**:

1. **利用目的の明示**
```
× 悪い例:
「個人情報を取得します」

○ 良い例:
「以下の目的で個人情報を利用します
 1. 商品の配送
 2. メールマガジンの配信
 3. アフターサービスのご案内」
```

2. **同意の取得**
```
× 悪い例:
勝手にメールアドレスを取得

○ 良い例:
□ メールマガジンの配信に同意する
  （チェックボックスで明示的に同意）
```

3. **安全管理**
```
必須対策
- パスワードは暗号化して保存
- アクセス権限を制限
- ログを記録
- 定期的なセキュリティ診断
```

4. **第三者提供の制限**
```
× 悪い例:
勝手に顧客リストを広告会社に渡す

○ 良い例:
「第三者への提供について同意いただけますか？
 提供先: ○○広告株式会社
 提供目的: ダイレクトメールの送付」
```

5. **開示請求への対応**
```
ユーザーから「自分の情報を見せて」と言われたら
→ 原則として開示が必要（本人確認後）
```

**プライバシーポリシーの必須項目**:
```
□ 事業者の名称・住所
□ 個人情報の利用目的
□ 第三者提供の有無
□ 開示請求の方法
□ 問い合わせ窓口
□ Cookie等の利用
□ 安全管理措置
```

**実務での会話例**:
- 「個人情報を扱うので、必ずプライバシーポリシーを掲載してください」
- 「パスワードは必ず暗号化して保存してください。平文はNG です」
- 「退会したユーザーの情報は、すぐに削除する必要があります」

### GDPR（EU一般データ保護規則）

**簡単に言うと**: EUの個人情報保護法（日本企業も対象になる）

**対象**:
- EU在住者の個人情報を扱う全ての企業
- 日本企業でも、EUユーザーがいれば対象

**日本の個人情報保護法との違い**:
```
┌──────────────┬──────────┬──────────┐
│項目          │日本      │EU(GDPR)  │
├──────────────┼──────────┼──────────┤
│罰金          │最大1億円 │最大30億円│
│対象          │日本国内  │EU在住者  │
│Cookie同意    │推奨      │必須      │
│データ削除権  │あり      │強い権利  │
└──────────────┴──────────┴──────────┘
```

**GDPR違反の罰金例**:
```
実際の罰金事例

Amazon: 746億円（2021年）
→ ターゲティング広告で同意を得ていない

Google: 56億円（2019年）
→ 広告のターゲティングが不透明

Facebook: 12億円（2019年）
→ パスワードを平文で保存
```

**GDPR対応で必要なこと**:
```
□ Cookie同意バナーの表示
□ データ削除権の対応
□ データポータビリティ（データのエクスポート機能）
□ 72時間以内の漏洩報告義務
□ プライバシーバイデザイン（設計段階からプライバシー配慮）
```

**Cookie同意バナーの例**:
```
┌───────────────────────────────────┐
│このサイトはCookieを使用します    │
│ □ 必須Cookie（同意不要）         │
│ □ 分析Cookie（同意必要）         │
│ □ 広告Cookie（同意必要）         │
│                                   │
│ [すべて同意] [設定] [拒否]       │
└───────────────────────────────────┘
```

**実務での会話例**:
- 「EU向けにサービスを提供するなら、GDPR対応が必須です」
- 「Cookie同意バナーを実装しないと、GDPR違反になります」
- 「データ漏洩が発生したら、72時間以内に報告義務があります」

## 6.3 セキュリティ脅威と対策

**簡単に言うと**: どんな攻撃があって、どう防ぐか

### フィッシング詐欺

**簡単に言うと**: 偽のサイトやメールで、パスワードやクレジットカード情報を盗む手口

**典型的な手口**:
```
STEP 1: 偽メールが届く
┌────────────────────────────────┐
│件名: [重要]アカウントが不正利用│
│されています！                  │
│                                │
│本文: すぐにログインして確認    │
│してください                    │
│→ https://amaz0n.com/login      │
│  （本物はamazon.com）          │
└────────────────────────────────┘
       ↓
STEP 2: 偽サイトに誘導
本物そっくりの偽ログイン画面
       ↓
STEP 3: 情報を入力してしまう
→ パスワードが盗まれる
```

**見分け方**:
```
フィッシングメールの特徴

1. 送信元のドメインが微妙に違う
   × amaz0n.com（ゼロ）
   ○ amazon.com（オー）

2. 緊急性を煽る文章
   「24時間以内に対応しないとアカウント停止」
   「不正利用されています」

3. 日本語がおかしい
   「あなたのアカウントは停止されました」
   「直ちに確認して下さい」

4. リンク先のURLが本物と違う
   マウスを乗せてURLを確認する
```

**対策**:
```
□ メールのリンクを直接クリックしない
□ 公式サイトをブックマークしておく
□ URLを必ず確認する
□ 二要素認証を有効化する
□ 不審なメールは開かない
```

**実務での会話例**:
- 「このメール、フィッシングじゃないですか？URLを確認してください」
- 「社員にフィッシング訓練を実施しましょう」

### ランサムウェア

**簡単に言うと**: ファイルを暗号化して、身代金を要求するウイルス

**攻撃の流れ**:
```
STEP 1: 感染
メール添付ファイルを開く
または、脆弱性のあるサーバーに侵入
       ↓
STEP 2: ファイルが暗号化される
すべてのファイルが開けなくなる
       ↓
STEP 3: 身代金要求
「ビットコインで300万円払えば復号キーを渡す」
       ↓
STEP 4: 支払っても...
復号キーが渡される保証はない
```

**被害事例**:
```
実際の被害

病院: 電子カルテが全て暗号化され、診療停止（2021年）
→ 復旧に数ヶ月、損害数億円

製造業: 工場の制御システムが暗号化され、生産停止（2022年）
→ 身代金1億円を支払い

自治体: 住民情報が暗号化され、行政サービス停止（2023年）
→ 復旧に半年
```

**対策**:
```
事前対策
□ 定期的なバックアップ（オフライン）
□ OSやソフトウェアの最新化
□ セキュリティソフトの導入
□ メール添付ファイルを開く際は注意
□ 権限の最小化（管理者権限で作業しない）

感染後の対応
□ すぐにネットワークから切断
□ 警察に通報
□ セキュリティ会社に相談
□ バックアップから復元
× 身代金は払わない（払っても復号される保証なし）
```

**実務での会話例**:
- 「ランサムウェア対策として、毎日バックアップを取りましょう」
- 「バックアップはオフラインで保存してください。オンラインだと一緒に暗号化されます」

### SQLインジェクション

**簡単に言うと**: Webサイトの入力欄から、データベースを不正に操作する攻撃

**攻撃の仕組み**:
```
通常のログイン
ユーザー名: tanaka
パスワード: pass123

SQL: SELECT * FROM users WHERE name='tanaka' AND password='pass123'
→ 正常にログイン

SQLインジェクション攻撃
ユーザー名: admin' OR '1'='1
パスワード: （空欄）

SQL: SELECT * FROM users WHERE name='admin' OR '1'='1' AND password=''
→ '1'='1'は常に真なので、パスワードなしでログインできる
```

**被害**:
```
SQLインジェクションでできること

× 全ユーザーの情報を取得
× データベースの内容を削除
× 管理者権限でログイン
× 個人情報の大量流出
```

**実際の被害事例**:
```
ECサイト: 10万件の個人情報漏洩（2020年）
→ クレジットカード情報含む

自治体: 住民の個人情報が漏洩（2019年）
→ 損害賠償数億円
```

**対策**:
```
開発側の対策
□ プリペアドステートメントを使う
□ 入力値のバリデーション
□ エラーメッセージを詳しく出さない
□ 最小権限の原則（データベースユーザーの権限を制限）

運用側の対策
□ WAF（Web Application Firewall）の導入
□ 定期的な脆弱性診断
□ ログの監視
```

**実務での会話例**:
- 「SQLインジェクション対策として、プリペアドステートメントを使ってください」
- 「入力値は必ずサニタイズしてください」

### XSS（クロスサイトスクリプティング）

**簡単に言うと**: Webサイトに悪意のあるスクリプトを埋め込む攻撃

**攻撃の仕組み**:
```
掲示板への投稿例

通常の投稿:
「こんにちは」
→ 画面に「こんにちは」と表示

XSS攻撃:
「<script>document.cookie を攻撃者に送信</script>」
→ このスクリプトが実行され、Cookieが盗まれる
```

**被害**:
```
XSSでできること

× Cookieの盗難（セッションハイジャック）
× 偽のログイン画面を表示
× フィッシングサイトへ誘導
× 個人情報の窃取
```

**対策**:
```
開発側の対策
□ 入力値のエスケープ処理
□ Content Security Policy (CSP)の設定
□ HTTPOnly属性のCookie
□ 入力値のバリデーション

例: エスケープ処理
入力: <script>alert('XSS')</script>
出力: &lt;script&gt;alert('XSS')&lt;/script&gt;
→ スクリプトとして実行されず、文字として表示
```

**実務での会話例**:
- 「ユーザー入力は必ずエスケープしてください」
- 「CSPヘッダーを設定して、XSS対策を強化しましょう」

### CSRF（クロスサイトリクエストフォージェリ）

**簡単に言うと**: ユーザーが意図しない操作を、勝手に実行させる攻撃

**攻撃の仕組み**:
```
STEP 1: ユーザーが銀行サイトにログイン中
       ↓
STEP 2: 攻撃者のサイトにアクセス
       ↓
STEP 3: 攻撃者のサイトに埋め込まれた悪意のあるリンク
<img src="https://bank.com/transfer?to=攻撃者&amount=100万円">
       ↓
STEP 4: 知らないうちに送金される
（ログイン中なので、本人の操作として処理される）
```

**対策**:
```
開発側の対策
□ CSRFトークンの利用
□ Referrerチェック
□ 重要な操作は再認証を要求

CSRFトークンの仕組み:
1. フォーム表示時にランダムな文字列（トークン）を生成
2. フォーム送信時にトークンを一緒に送る
3. サーバーでトークンが正しいか確認
→ 攻撃者のサイトからは正しいトークンを送れない
```

**実務での会話例**:
- 「重要な操作には必ずCSRFトークンを使ってください」
- 「送金など重要な操作では、パスワードの再入力を求めましょう」

## 6.4 セキュリティチェックリスト

**簡単に言うと**: サービス公開前に必ず確認すべき項目

### 最低限のセキュリティチェック

```
□ パスワード
  □ パスワードは暗号化して保存（bcrypt、Argon2等）
  □ パスワードの強度チェック（8文字以上、複雑さ）
  □ パスワードリセット機能
  □ 二要素認証の実装

□ 個人情報保護
  □ プライバシーポリシーの掲載
  □ SSL/TLS（https）の導入
  □ 個人情報の暗号化
  □ アクセスログの記録

□ Web攻撃対策
  □ SQLインジェクション対策
  □ XSS対策（エスケープ処理）
  □ CSRF対策（トークン）
  □ クリックジャッキング対策

□ アクセス制御
  □ 適切な権限管理
  □ セッションタイムアウト
  □ ログイン試行回数の制限
  □ IPアドレス制限（管理画面）

□ データ管理
  □ 定期的なバックアップ
  □ バックアップからの復元テスト
  □ 不要なデータの削除
  □ ログの保存期間を設定

□ 開発・運用
  □ 最新のセキュリティパッチ適用
  □ 不要なポートを閉じる
  □ エラーメッセージを詳しく出さない
  □ セキュリティ診断の実施
```

**実務での会話例**:
- 「リリース前に、このチェックリストを確認してください」
- 「セキュリティ診断を外部企業に依頼しましょう」
